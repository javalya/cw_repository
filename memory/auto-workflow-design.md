# INSIGHT AI STUDIO 自运转工作流设计

**版本**: v1.0  
**目标**: 公司自动化运转，刘总仅需关键决策  
**设计日期**: 2026-03-20

---

## 核心循环

```
需求发现 → 机会评估 → 决策审批 → 产品开发 → 测试发布 → 运营反馈 → 迭代优化
     ↑                                                              │
     └──────────────────────────────────────────────────────────────┘
```

---

## 第一阶段：需求发现（全自动）

### 触发器
- **时间触发**: 每周一/周四 10:00
- **事件触发**: 社交媒体热点、竞品动态

### 执行流程
```
┌─────────────────────────────────────────┐
│  Needs Research Agent 自动运行           │
│  1. 爬取 Reddit/小红书/即刻需求表达      │
│  2. 统计痛点频率和趋势                   │
│  3. 扫描现有竞品                         │
│  4. 生成《需求调研周报》                 │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  Product Analyst 分析                   │
│  1. 评估市场规模                         │
│  2. 分析竞争格局                         │
│  3. 计算可行性评分                       │
│  4. 输出《机会评估报告》                 │
└─────────────────────────────────────────┘
                    ↓
          推送给刘总（决策点1）
```

### 输出物
- 《需求调研周报》（自动发送到飞书）
- 《机会评估报告》（TOP 3机会推荐）

### 刘总决策
- **输入**: 调研报告 + 机会评估
- **决策**: 选择要开发的方向 / 暂缓 / 需要更多信息
- **时间**: 24小时内回复
- **默认**: 如不回复，选择评分最高的机会自动进入下一阶段

## 第二阶段：产品规划（半自动）

### 触发条件
- 刘总批准某个方向
- 或自动选择评分>8分的机会

### 执行流程
```
┌─────────────────────────────────────────┐
│  Content Lead 制定产品策略               │
│  1. 确定产品定位                         │
│  2. 定义目标用户                         │
│  3. 制定差异化策略                       │
│  4. 输出《产品策略文档》                 │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  Tech Architect 技术评估                 │
│  1. 技术可行性分析                       │
│  2. 技术选型建议                         │
│  3. 工作量估算                           │
│  4. 输出《技术方案概要》                 │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  Growth Hacker 增长预估                  │
│  1. 获客渠道分析                         │
│  2. 预估获客成本                         │
│  3. 制定冷启动策略                       │
│  4. 输出《增长方案》                     │
└─────────────────────────────────────────┘
                    ↓
          推送给刘总（决策点2）
```

### 输出物
- 《产品策略文档》
- 《技术方案概要》
- 《增长方案》
- 《项目启动建议书》（汇总）

### 刘总决策
- **输入**: 产品+技术+增长方案
- **决策**: 启动项目 / 调整方案 / 放弃
- **输出**: 确认MVP范围、预算、时间预期

## 第三阶段：产品开发（全自动）

### 触发条件
- 刘总批准项目启动

### 执行流程

#### 3.1 设计阶段（并行）
```
UI/UX Designer
├── Day 1-2: 信息架构 + 低保真原型
├── Day 3-4: 高保真设计稿
├── Day 5: 设计规范 + 切图交付
└── 输出: Figma设计稿 + 设计规范文档
```

#### 3.2 技术方案阶段（并行）
```
Tech Architect
├── 输出: 详细技术架构文档
├── 定义: API接口规范
├── 制定: 数据库设计
└── 分配: 开发任务给各工程师
```

#### 3.3 开发阶段（并行）
```
Week 1-2:
├── Mobile Dev Lead
│   ├── Flutter项目搭建
│   ├── 基础框架开发
│   └── 核心页面实现
├── Backend Engineer
│   ├── Firebase配置
│   ├── 数据模型设计
│   └── API接口开发
└── DevOps Engineer
    ├── CI/CD流水线搭建
    ├── 自动化测试配置
    └── 监控告警设置

Week 3:
├── 功能联调
├── 集成测试
└── Bug修复
```

#### 3.4 自动化检查点
```
每次代码提交:
├── GitHub Actions自动构建
├── 自动化测试运行
├── 代码质量检查
└── 失败则阻止合并

每日晚上:
├── 自动打包测试版本
├── 上传到TestFlight（iOS）
├── 上传到Firebase（Android）
└── 发送测试报告
```

### 进度同步
- **每日**: 自动汇总开发进度到飞书群
- **每周**: 生成《开发周报》
- **里程碑**: 达到关键节点自动通知刘总

---

## 第四阶段：测试发布（半自动）

### 触发条件
- 开发完成（所有功能实现）
- 自测通过（开发者自测）

### 执行流程
```
┌─────────────────────────────────────────┐
│  QA/Tester 测试阶段                      │
│  1. 功能测试                             │
│  2. 兼容性测试                           │
│  3. 性能测试                             │
│  4. 输出《测试报告》                     │
└─────────────────────────────────────────┘
                    ↓
         如有Bug → 自动分配给开发者修复
         如通过 → 进入发布流程
                    ↓
┌─────────────────────────────────────────┐
│  DevOps Engineer 发布准备                │
│  1. 生产环境配置                         │
│  2. 应用商店材料准备                     │
│  3. 发布检查清单确认                     │
│  4. 灰度发布配置                         │
└─────────────────────────────────────────┘
                    ↓
          推送给刘总（决策点3）
```

### 输出物
- 《测试报告》（通过率、Bug统计）
- 《发布准备清单》
- 《应用商店素材包》

### 刘总决策
- **输入**: 测试报告 + 发布准备情况
- **决策**: 发布 / 修复后发布 / 暂缓
- **选项**: 直接发布 / 灰度发布（先给100人）

---

## 第五阶段：运营反馈（全自动）

### 触发条件
- App发布后自动启动

### 执行流程
```
每日自动:
├── 数据采集
│   ├── 下载量统计
│   ├── 活跃用户数
│   ├── 留存率
│   ├── 崩溃率
│   └── 用户评价
├── 数据分析
│   ├── Growth Hacker分析增长数据
│   ├── Product Analyst分析用户行为
│   └── 生成《运营日报》
└── 自动推送飞书群

每周自动:
├── 生成《运营周报》
├── 对比上周数据
├── 识别异常指标
└── 提出优化建议

每月自动:
├── 生成《月度复盘报告》
├── 收入统计分析
├── 用户画像更新
└── 下月运营计划建议
```

### 异常告警
```
自动监控指标:
├── 崩溃率 > 1% → 立即告警 → 自动回滚评估
├── 日活下降 > 20% → 日报警 → 分析原因
├── 差评率 > 10% → 日报警 → 问题归类
└── 收入异常 → 周报警 → 深度分析
```

## 第六阶段：迭代优化（半自动）

### 触发条件
- 收集到足够的用户反馈
- 数据出现异常
- 定期（每月）主动审视

### 执行流程
```
┌─────────────────────────────────────────┐
│  Product Analyst 数据分析                │
│  1. 用户反馈归类                         │
│  2. 功能使用热力图                       │
│  3. 流失原因分析                         │
│  4. 输出《优化建议清单》                 │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  Tech Architect 技术评估                 │
│  1. 技术债务评估                         │
│  2. 性能瓶颈分析                         │
│  3. 架构优化建议                         │
│  4. 输出《技术优化方案》                 │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  Content Lead 内容规划                   │
│  1. 内容更新计划                         │
│  2. 运营活动策划                         │
│  3. 输出《迭代计划》                     │
└─────────────────────────────────────────┘
                    ↓
          推送给刘总（决策点4）
```

### 刘总决策
- **输入**: 优化建议 + 数据表现
- **决策**: 进入下一轮迭代 / 维持现状 / 停止项目
- **输出**: 确认迭代优先级

---

## 决策点汇总

| 决策点 | 触发条件 | 决策内容 | 默认行为 | 时限 |
|--------|----------|----------|----------|------|
| **DP1** | 每周需求报告 | 选择开发方向 | 选最高评分 | 24h |
| **DP2** | 产品方案完成 | 批准启动项目 | 自动启动 | 48h |
| **DP3** | 测试通过 | 批准发布 | 灰度发布 | 24h |
| **DP4** | 月度复盘 | 迭代/维持/停止 | 进入迭代 | 48h |

**说明**: 刘总不回复则按默认行为自动执行

## 自动化触发器配置

### Cron任务（OpenClaw配置）
```yaml
# 需求调研
cron_jobs:
  - name: needs_research_weekly
    schedule: "0 10 * * 1"  # 每周一10点
    action: run_needs_agent
    notify: feishu

  - name: daily_sync
    schedule: "0 22 * * *"  # 每天22点
    action: generate_daily_report
    notify: feishu

  - name: weekly_review
    schedule: "0 10 * * 5"  # 每周五10点
    action: generate_weekly_report
    notify: feishu

  - name: monthly_review
    schedule: "0 10 1 * *"  # 每月1号10点
    action: generate_monthly_report
    notify: feishu
```

### 事件触发器
```yaml
event_triggers:
  - event: code_push
    action: run_ci_pipeline
    
  - event: test_complete
    condition: pass_rate > 90%
    action: notify_release_ready
    
  - event: app_crashed
    condition: crash_rate > 1%
    action: alert_devops + notify_architect
    
  - event: user_review
    condition: rating < 3
    action: analyze_feedback + notify_product
```

## 风险管控

### 自动止损机制
```
发布后监控:
├── 1小时内崩溃率>5% → 自动回滚
├── 24小时内评分<2星 → 暂停推广
├── 7日留存<10% → 产品Review会议
└── 收入连续3月为负 → 刘总特别决策
```

### 人工介入触发
```
需要刘总介入:
├── 技术方案存在争议
├── 预算超支>20%
├── 竞品突然发布同类产品
├── 法律/合规风险
└── 团队Agent之间冲突
```

## 成功指标

### 运转效率指标
- 需求→上线周期: < 6周
- 刘总决策耗时: < 24h
- 自动化率: > 80%
- 发布成功率: > 95%

### 业务指标
- 产品迭代频率: 2周/次
- Bug修复时效: P0<4h, P1<24h
- 用户反馈响应: < 48h
- 数据报告延迟: < 4h
