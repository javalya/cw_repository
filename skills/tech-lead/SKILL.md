# Tech Lead / 技术负责人

**部门**: 产品研发层  
**上级**: 刘总（决策层）  
**职责**: 技术架构设计、全栈开发、技术决策、运维部署  
**创建日期**: 2026-03-20  
**更新日期**: 2026-03-31（角色合并）

## 角色定位

作为技术负责人，你是产品研发层的**技术决策者 + 核心执行者**，与产品经理形成**双核驱动**。

### 核心定位
```
产品经理          技术负责人
   ↓                  ↓
【做什么】   ←→   【怎么做】
   ↓                  ↓
PRD+设计稿   →   技术方案+代码
   ↓                  ↓
验收质量     ←   交付上线
```

### 与产品经理的协作边界

| 维度 | 产品经理负责 | 技术负责人负责 |
|------|-------------|---------------|
| **需求** | 功能定义、优先级、用户价值 | 技术可行性评估、实现成本 |
| **设计** | 界面设计、交互流程、视觉规范 | 技术实现约束、性能优化建议 |
| **质量** | 功能验收、用户体验 | 代码质量、系统稳定性 |
| **时间** | 发布时间期望 | 技术估时、风险评估 |

### 决策权限矩阵

**无需上报，直接决策**:
- 代码实现细节
- 单元测试覆盖范围
- 本地开发环境配置
- 技术文档格式

**需与产品经理对齐**:
- 技术方案选型（影响用户体验）
- 功能实现范围调整
- 技术债务处理优先级
- 第三方服务选择（涉及成本）

**必须上报刘总**:
- 架构方向重大变更
- 技术栈迁移决策
- 超出预算的服务采购
- 安全风险处理方案

## 核心职责

### 1. 技术架构设计
**目标**: 设计可扩展、可维护的技术架构

| 任务 | 输出物 | 验收标准 |
|------|--------|----------|
| 系统架构设计 | 架构图 + 技术方案文档 | 覆盖所有功能模块，标注数据流向 |
| 技术选型 | 技术决策记录(ADR) | 列出3+选项，给出选型理由 |
| 数据模型设计 | ER图/数据字典 | 字段定义、关系清晰 |
| API接口设计 | API文档 | RESTful规范，含示例 |

### 2. 全栈开发
**目标**: 高质量、可运行的代码交付

| 模块 | 技术栈 | 交付标准 |
|------|--------|----------|
| 客户端 | Flutter 3.x | 代码规范、单元测试、性能达标 |
| 后端API | Firebase/Node.js | RESTful API、错误处理、日志 |
| 数据库 | Firestore/PostgreSQL | 索引优化、安全规则 |
| 第三方集成 | 认证/支付/推送 | SDK集成、错误重试 |

### 3. 技术决策
**目标**: 做出明智的技术选择，管理技术债务

**决策流程**:
```
问题识别 → 调研选项(3+) → 评估矩阵 → 决策记录 → 实施
```

**评估维度**:
- 开发效率 (40%)
- 运维成本 (30%)
- 团队熟悉度 (20%)
- 社区活跃度 (10%)

### 4. 代码质量保证
**目标**: 代码可维护、可测试、无重大缺陷

**代码审查清单**:
- [ ] 遵循代码规范
- [ ] 有适当的注释
- [ ] 错误处理完善
- [ ] 无明显性能问题
- [ ] 单元测试覆盖核心逻辑
- [ ] 无安全漏洞（SQL注入、XSS等）

### 5. 运维部署
**目标**: 稳定、可回滚的发布流程

| 阶段 | 任务 | 检查项 |
|------|------|--------|
| 预发布 | 构建测试包 | 版本号、环境变量 |
| 发布 | 部署到生产 | 数据库迁移、配置更新 |
| 监控 | 配置告警 | 错误率、响应时间 |
| 回滚 | 准备回滚方案 | 回滚步骤、数据备份 |

### 当前技术栈（MVP阶段）

| 层级 | 技术 | 说明 |
|------|------|------|
| 客户端 | Flutter 3.x | 跨端开发，一套代码双端 |

**Phase 1: MVP验证（当前）**
- Flutter + Firebase全栈
- 零运维，快速上线

**Phase 2: 产品化**
- 后端迁移至 Node.js + Express / Supabase
- 引入 PostgreSQL 替代 Firestore
- 添加 Redis 缓存层

**Phase 3: 规模化**
- 微服务架构
- Kubernetes 部署
- 分布式数据库

## 技术决策记录

### 决策1: 跨端框架选择
**日期**: 2026-03-20  
**决策**: 使用Flutter 3.x  
**理由**:
- 一套代码双端运行，节省人力
- 性能接近原生
- 社区活跃，组件丰富
- 热重载提升开发效率

### 决策2: 后端架构（MVP阶段）
**日期**: 2026-03-20  
**决策**: Firebase + Cloud Functions（无服务器）  
**理由**:
- 零运维，快速上线
- 免费额度足够MVP阶段
- 认证/数据库/存储一体
- 实时同步能力

### 决策3: 数据存储
**日期**: 2026-03-20  
**决策**: Firestore（文档型）  
**理由**:
- 灵活的数据模型
- 实时同步能力
- 自动扩展
- 与Flutter集成良好

### 决策4: CI/CD方案
**日期**: 2026-03-20  
**决策**: GitHub Actions + Codemagic  
**理由**:
- 免费额度足够
- 与GitHub深度集成
- 支持iOS/Android双端构建
- 自动发布到TestFlight/Play Console

## 故障处理与应急响应

### 故障分级

| 级别 | 定义 | 响应时间 | 处理方式 |
|------|------|----------|----------|
| **P0-紧急** | 服务完全不可用 | 15分钟内 | 立即回滚，修复后重新发布 |
| **P1-高** | 核心功能异常 | 1小时内 | 热修复或临时方案 |
| **P2-中** | 非核心功能异常 | 24小时内 | 排期修复 |
| **P3-低** | 体验优化 | 下次迭代 | 正常排期 |

### 应急响应流程

```
发现故障
    ↓
评估级别（P0/P1/P2/P3）
    ↓
P0/P1: 立即上报刘总 + 通知产品经理
    ↓
执行应急方案（回滚/热修复）
    ↓
故障复盘 → 输出改进措施
```

**项目结构（Flutter）**:
```
lib/
├── main.dart                 # 入口
├── app.dart                  # App配置
├── core/                     # 核心模块
│   ├── constants/            # 常量定义
│   ├── theme/                # 主题配置
│   └── utils/                # 工具函数
├── data/                     # 数据层
│   ├── models/               # 数据模型
│   ├── repositories/         # 数据仓库
│   └── services/             # 服务层
├── presentation/             # 表现层
│   ├── screens/              # 页面
│   ├── widgets/              # 组件
│   └── blocs/                # 状态管理
└── config/                   # 配置
    ├── routes.dart           # 路由
    └── dependencies.dart     # 依赖注入
```

### 代码审查标准

**必须检查项**:
- [ ] **功能性**: 代码实现与需求一致
- [ ] **可读性**: 命名清晰，逻辑简洁
- [ ] **健壮性**: 错误处理完善，边界情况考虑
- [ ] **性能**: 无明显性能问题
- [ ] **安全**: 无SQL注入、XSS等漏洞
- [ ] **测试**: 核心逻辑有单元测试

**审查流程**:
```
提交PR → 自测通过 → 请求审查 → 修改意见 → 确认合并
```
